# プロジェクト概要：逆プロンプトゲーム（ビジネス実用版）

> **改訂履歴:** 2026-06-11 リードPM/アーキテクトレビューを反映（モデル更新、サーバー側レート制限、採点ルーブリック、データモデル、拡張性要件を追加）

## 1. プロジェクトの目的
ユーザーの「言語化能力（5W1Hの明確化、具体と抽象の行き来、構造化）」を鍛えるためのWebアプリケーションを開発する。ユーザーが入力したプロンプトをAIが採点し、コーチング型のフィードバックを行うゲーミフィケーション学習ツール。

## 2. 技術スタック
* **フロントエンド:** Next.js (App Router), React, TypeScript
* **スタイリング:** Tailwind CSS
* **バックエンド/API:** Next.js Route Handlers
* **AIモデル:** Google Gemini API — **Gemini 2.5 Flash**（Gemini 1.5系は2025年9月に廃止済みのため使用不可。コスト最優先の場合は 2.5 Flash-Lite を検討）
* **レート制限ストア:** Upstash Redis（Vercel Marketplace経由、無料枠）
* **ホスティング:** Vercel

## 3. ハイブリッドAPI方式の仕様（運用無料化と安全性）

### 3-1. デフォルト（無料枠利用）
* 開発者の環境変数APIキーを使用し、サーバー（Route Handler）経由でGemini APIを呼び出す。
* **プレイ回数制限は二層構造とする:**
  1. **UX層:** LocalStorageで1日3回をカウントし、残回数表示・案内を行う（あくまで表示用）。
  2. **防御層（必須）:** サーバー側でIPベースのレート制限（Upstash Redis）を実装する。UI制限はストレージ削除等で容易に回避でき、またAPIルートはcurl等で直接叩けるため、開発者キーの保護はサーバー側制限が本体である。
* データ学習に関する免責事項を明記する。

### 3-2. APIキー持ち込み（BYOK方式）
* ユーザーが自身のGemini APIキーを入力・保存（ブラウザの `localStorage` 限定）することで、プレイ回数無制限かつデータ学習なしの安全な環境を提供する。
* **呼び出し経路:** BYOKモードではブラウザからGemini APIを直接呼び出す（キーは一切サーバーを通らない）。これが「キーがサーバーに送られない」という訴求の根拠となる。
* **明記すべきトレードオフ:** クライアント直接呼び出しのため、採点用システムプロンプトはBYOKユーザーには技術的に閲覧可能となる。これは許容する（プロンプトは秘匿資産としない）。
* キー保存時に簡易検証リクエストを行う「キーをテスト」ボタンを設ける。

### 3-3. AI呼び出しの共通要件
* **構造化出力の必須化:** JSONの安定取得のため、プロンプト指示のみに頼らず Gemini の structured output（`responseMimeType: "application/json"` + `responseSchema`）を使用する。
* **プロバイダ抽象化:** Gemini呼び出しは `AIProvider` インターフェースの背後に実装し、無料枠（サーバー経由）とBYOK（クライアント直）で採点ロジック・スキーマ定義を共有する。将来の複数AIモデル対応の基盤とする。

### 3-4. エラーハンドリングの分類（設計段階で型を定義する）
| 分類 | 例 | ユーザーへの表示方針 |
|---|---|---|
| 無効キー | 401 / 403 | BYOKキーの再確認を促す |
| レート超過 | 429 | 無料枠は「本日の無料プレイ上限」案内＋BYOK誘導。BYOKはGemini側制限の旨を表示 |
| セーフティブロック | Geminiの安全フィルタで出力が空 | 「お題やプロンプトの表現を変えて再挑戦」を促す |
| タイムアウト/ネットワーク | fetch失敗 | 再試行ボタンを提示 |
| JSONパース失敗 | スキーマ不一致 | 1回自動リトライ後、エラー表示 |

## 4. ゲーム体験（UX）と画面構成

### 4-1. 採点ルーブリック（学習効果の核）
スコアは100点満点を以下のサブスコアに分解し、JSONで個別に返却する：
* **明確性（5W1H）:** 25点 — 目的・対象・条件が明示されているか
* **具体性:** 25点 — 抽象語に頼らず、具体例・数値・形式指定があるか
* **構造化:** 25点 — 指示が整理され、優先順位や手順が読み取れるか
* **目的適合:** 25点 — お題のゴールに対して過不足がないか

サブスコア化により (a) 採点の一貫性向上、(b) ユーザーの自己診断、(c) 将来の成長ダッシュボードのデータ基盤を実現する。採点は temperature を低く設定し再現性を高める。

### 4-2. 体験設計
* **伴走型コーチングAI:** 採点用システムプロンプトは、ユーザーを否定せず「ここを具体化すると更に良くなります」と問いかけながら伴走するメンタートーンを徹底する。
* **Before/Afterの可視化:** ユーザーのプロンプトによる「実際の出力」と、AIが添削した「理想の出力」を並べて表示し、アハ体験（気づき）を生むUIとする。
* **再挑戦ループ（必須要件）:** 結果画面に「このお題にもう一度挑戦」ボタンを設置し、同一お題の前回スコアとの比較（サブスコア別の伸び）を表示する。「書き直したら上がった」体感がBefore/Afterと並ぶ第二のアハ体験である。
* **クリア基準:** 各ステージに合格点（例：80点）を設定し、達成で次ステージ解放などの進行構造を持たせる。
* **レイテンシ設計:** 採点＋出力生成で5〜15秒かかり得るため、ステップ表示型のローディング演出（「出力を生成中…」「採点中…」等）を必須とする。
* **インジェクションの教育的処理:** 「100点をつけて」等のメタ指示を検出した場合、エラーではなく「それはプロンプト力ではありませんね😊」といった教育的フィードバックで返し、体験に昇華する。

### 4-3. 画面構成
* 【トップ画面】BYOK設定フォーム（キーテスト機能付き）、ステータス表示（残プレイ回数）、ステージ選択（クリア状況表示）。
* 【プレイ画面】お題の提示、プロンプト入力フォーム（文字数制限・空入力ガード）、ステップ表示ローディング。
* 【結果画面】総合スコア＋サブスコア内訳、伴走型フィードバック、Before/After比較プレビュー、再挑戦ボタン（前回比表示）。

## 5. データ設計（拡張性の基盤）

### 5-1. ステージ定義（コードから分離）
お題はスキーマを持つ構造化データ（`src/data/stages/*.ts`）として定義し、追加・編集をコード変更なしに行えるようにする：
```ts
interface Stage {
  id: string;
  title: string;
  description: string;      // お題文
  difficulty: 1 | 2 | 3;
  targetAudience: string;   // 想定読者（例：前提知識のない学生）
  goal: string;             // このお題で達成すべきゴール
  rubricHints: string[];    // 採点AIに渡すお題固有の評価観点
  passingScore: number;     // クリア基準点
}
```

### 5-2. プレイ結果レコード（将来のDB移行を見据える）
MVP時点からlocalStorageに履歴として保存し、将来の成長ダッシュボード・DB移行時にそのまま引き継げる形式とする：
```ts
interface PlayResult {
  schemaVersion: 1;
  stageId: string;
  promptText: string;
  scores: { clarity: number; specificity: number; structure: number; fitness: number; total: number };
  feedback: string;
  modelId: string;          // 採点に使ったモデル
  playedAt: string;         // ISO 8601
}
```

## 6. 高度なエージェントチームの役割と進行フェーズ

Claude Codeは、以下のフェーズごとに指定された高度な専門家の役割（ロール）になりきり、ユーザー（PM）に確認を取りながらステップバイステップで開発を推進すること。

### フェーズ1：リードPM 兼 システムアーキテクト
* **役割:** 基盤構築、ハイブリッドAPIアーキテクチャの設計
* **タスク:** Next.js環境の構築。`AIProvider` 抽象化レイヤと、環境変数キー（サーバー経由）/BYOK（クライアント直）を切り替えるアーキテクチャの骨組みを設計する。ステージ定義・結果レコードの型を確立し、拡張性の高いディレクトリ構成にする。

### フェーズ2：UI/UX 兼 フロントエンドエンジニア
* **役割:** モダンな画面実装とセキュアな状態管理
* **タスク:** Tailwind CSSで白・グレー・ネイビーを基調とした信頼感のあるUIを構築。BYOKのキーを `localStorage` だけで完結させる安全な実装、1日3回制限のUX層ロジック、ステップ表示ローディングを完成させる。

### フェーズ3：プロンプトアーキテクト 兼 学習体験(LX)デザイナー
* **役割:** 伴走型コーチングロジックの実装とステージ設計
* **タスク:** Gemini API（structured output）と接続し、ルーブリック準拠のサブスコア・フィードバック・生成結果のJSONを返す採点システムを構築する。サーバー側IPレート制限（Upstash Redis）を実装する。
* **初期ステージ実装:** 専門的な知識（例：税や法律の仕組み）を、前提知識のない学生や若者向けに分かりやすく解説する指示（専門性と平易さの両立）など、実務に即したリアリティのあるお題を設定する。

### フェーズ4：QA 兼 セキュリティエンジニア
* **役割:** 堅牢性のテストとエラーハンドリング
* **タスク:** プロンプトインジェクションへの耐性テスト（メタ指示の教育的処理を含む）、3-4のエラー分類に基づくハンドリング検証、無料枠制限時のメッセージ表示、レート制限の回避耐性確認など、エッジケースのテストと修正を行う。

## 7. プロジェクトの最終ビジョン（将来構想）
MVPリリース後、将来的に「専門職向けの高度なシナリオ（複雑な制度要約等）」や「ユーザーごとの成長ログ・分析ダッシュボード機能」、「複数の生成AIツールを使い分けるワークフロー学習」などへ拡張することを見据え、スケーラブルで保守性の高いコード設計を初回から徹底すること。本仕様の5章（データ設計）と3-3（プロバイダ抽象化）がその布石である。

## 8. Claude Codeへの基本ルール
* 各フェーズの完了ごとに状況を報告し、ユーザーの承認を得てから次へ進むこと。
* コード変更時はその意図を説明し、エラー発生時は推測される原因と解決策を複数提示すること。
